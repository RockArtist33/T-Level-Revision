    # 2.8.1 - Understand the fundamental importance of testing for all components
    • Software
      • Software ensures that a piece of software performs as expected and meets the requirements and specifications set forth by the developers. Testing can help identify and fix bugs and other issues before the software is released to users. This can prevent user frustration and improve the overall quality of the software. Additionally, testing can help ensure that the software is reliable and able to handle a variety of scenarios and inputs. This is especially important for mission-critical software, where even a small error can have serious consequences. Overall, testing is a crucial step in the software development process that helps ensure that the final product is of high quality and meets the needs of users.
    • Hardware
      • It allows designers and manufacturers to ensure that a piece of hardware is reliable, functional, and safe to use. Testing can help identify potential problems or issues with a design, and allow engineers to make changes or improvements before a product is released to the market. Without proper testing, a piece of hardware could fail or malfunction, leading to potentially dangerous situations or costly recalls. Therefore, testing is an essential part of the hardware development process and helps ensure that a product will work as intended and meet the needs of its users.
    • Data
      • It helps ensure that the data we are working with is accurate and reliable. By conducting tests on our data, we can identify any potential issues or errors and address them before using the data for analysis or making decisions. This is especially important in fields like healthcare and finance, where even small errors in the data can have significant consequences. Testing also helps us to gain confidence in our data and the results that we are able to generate from it. Without testing, we may be basing our decisions on faulty or incomplete data, which can lead to incorrect or misleading conclusions.
    • Interfaces
      • The user interface is the primary way that users interact with a software application, and if it is not functioning correctly, it can lead to a poor user experience. Testing an interface ensures that it is working as intended and that it is easy for users to navigate and use. It also helps to identify and fix any bugs or issues before the software is released to the public. This can save time and resources in the long run by preventing the need for costly and time-consuming updates and fixes after the software has been released.
    • Final product
      • It ensures that the product is of high quality and is free of defects or bugs. This is especially important for products that will be used by customers, as it can help to improve customer satisfaction and build trust in the brand. This can help to ensure that the product is fit for its intended purpose and meets the needs of the customer, testing is a crucial step in the development process that helps to ensure that the final product is of high quality, free of defects, and meets the needs of the customer.
    # 2.8.2 - Understand the use of testing and quality assurance methodologies to seek out problems and issues
    • concept testing
      • Concept testing (to be distinguished from pre-test markets and test markets which may be used at a later stage of product development research) is the process of using surveys (and sometimes qualitative methods) to evaluate consumer acceptance of a new product idea prior to the introduction of a product to the market.
    • unit testing
      • In computer programming, unit testing is a software testing method by which individual units of source code, sets of one or more computer program modules together with associated control data, usage procedures, and operating procedures are tested to determine whether they are fit for use.
    • boundary testing
      • Boundary Testing is value analysis, a type of black box or specification based testing technique in which tests are performed using the boundary values. The goal of boundary testing is to ensure that the system can handle input values that are at the edge of the acceptable range these are called extremes or edge cases.
    • integration testing
      • Integration testing is the phase in software testing in which individual software modules are combined and tested as a group. Integration testing is conducted to evaluate the compliance of a system or component with specified functional requirements.
    • performance testing
      • Performance testing is in general a testing practice performed to determine how a system performs in terms of responsiveness and stability under a particular workload. It can also serve to investigate, measure, validate or verify other quality attributes of the system, such as scalability.
    • system testing
      • System testing is testing conducted on a complete integrated system to evaluate the system’s compliance with its specified requirements. System testing takes, as its input, all of the integrated components that have passed integration testing. The purpose of integration testing is to detect any inconsistencies between the units that are integrated together.
    • acceptance and usability testing
      • Acceptance testing is a test conducted to determine if the requirements of a specification or contract are met. It may involve chemical tests, physical tests, or performance tests.
    • regression testing
      • Regression testing (rarely, non-regression testing) is re-running functional and non-functional tests to ensure that previously developed and tested software still performs after a change. If not, that would be called a regression.
    • load/stress testing.
      • Load testing is the process of putting demand on a structure or system and measuring its response.
    # 2.8.3 - Understand how automated and functional testing tools can be applied to test digital systems and code.

    Automated and functional and both

    These tools can be used to test digital systems and code in a number of ways. Automated testing tools can be used to automate the execution of tests, which can help to save time and reduce the need for manual testing. Functional testing tools, on the other hand, can be used to test the functionality of a system or application to ensure that it behaves as expected.

    One way in which automated testing tools can be used is to create and run test scripts that exercise the different features and functions of a digital system or application. These scripts can be designed to test different aspects of the system, such as its user interface, performance, or reliability. The test scripts can then be run automatically, and the results can be recorded and analysed to identify any defects or issues.

    Functional testing tools, on the other hand, can be used to test the functionality of a system or application by simulating the actions of a user and checking that the system behaves as expected. For example, a functional testing tool might be used to simulate a user clicking on a button in a user interface, and then check that the system responds appropriately.

    Overall, automated and functional testing tools can be used to quickly and efficiently test digital systems and code, helping to ensure that they are of high quality and free of defects. These tools can be used in combination with other testing techniques to provide a comprehensive view of the system’s behaviour.

    # 2.8.5 - Understand how to construct an effective test plan, including

    • identifying tests to be carried out & describing the purpose of the identified test

      • concept testing: just asking others opinions. comparing testing

      • boundary testing: checks the extreme ends of input in software.

      • integration testing: testing the individual modules as combined and tested as a group

      • performance testing: test how well it actually performance and see if someone can enter it i.e. hacking .

      • system testing: testing the fully integrated software product. testing end to end system.

      • regression testing: tests to check if a program has not been broken by a change in code.

      • load tests:help you understand how a a system behavior under an expected load

      • stress test:pushes the system to its upper limit using a load beyond the expected maximum

    • identifying test data to be used

      • valid
        • Sensible data that the program should accept and should not cause the program to break if all things are working correctly
      • valid extreme
        • Data that falls at the edges/boundaries of the accepted possible range
      • invalid
        • Data that the system/application should not accept and should cause the program to have an error which needs handling
      • invalid extreme
        • Data that is invalid but only just at the extremes
      • erroneous
        • Data which is not valid and does meet the correct format and should be rejected

    • describing the expected results.

      • Expected Results
        • What results are you expecting if the program was working as it should?
      • Actual Results
        • What happens when the test is run? This is where you will see if the functionality you are testing works or not as you compare it to the expected result and if they do not match, the test is classed as failing and requires fixing
      • Comments/Actions
        • What were the actions taken to fix the error if they did occur? Any general comments about the test and how it went and if it was a success or failure
      • Making Changes
        • When making the actual change for your failed code, you must ensure you always re run the tests completely to ensure your fix did not break any other part of the code base. This is called regression testing
